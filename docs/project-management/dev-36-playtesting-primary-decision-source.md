# DEV-36 — Playtesting как основной источник решений

## Что есть сейчас
- Playtest упомянут в roadmap, но не был связан с единым наблюдаемым циклом «сессия → метрики → приоритизация». 
- В проекте уже есть понятные UX-ориентиры (time-to-fun, ясность цели, отсутствие downtime), но не было технической фиксации ключевых поведенческих сигналов для каждой попытки.
- Решения по UX-правкам рисковали приниматься по субъективному ощущению разработчика, а не по частоте проблемы у игроков.

## Что должно быть в хорошем продукте
- Playtesting становится регулярным циклом для DEV-28:
  - 5–10 наблюдаемых сессий;
  - молчаливое наблюдение за игроком;
  - фиксация точек: где игрок теряется, где скучает, где получает «непонятный» урон.
- Для каждой сессии собираются базовые метрики поведения:
  - time-to-fun (время до первого осмысленного боевого контакта),
  - время до исхода (первая победа/поражение/выход),
  - частота пауз,
  - частота выхода в меню из паузы.
- Приоритизация backlog выполняется по частоте повторения проблемы между сессиями:
  - High: проблема встречается у 40%+ тестов,
  - Medium: 20–39%,
  - Low: <20%.

## Элегантная реализация в MVP

### 1) Лёгкая in-client телеметрия без внешней инфраструктуры
- Добавлен модуль `playtestTelemetry` с хранением последних 20 сессий в `localStorage`.
- Модуль не требует backend и позволяет сразу запускать серию тестов на DEV-стенде.

### 2) Интеграция в текущий flow сцен
- Старт сессии: при старте игры из `MenuScene`.
- Time-to-fun: при первом подтверждённом боевом событии в `LevelScene`.
- Частота паузы: при каждом входе в `PauseScene` из Level/Boss.
- Выход игрока: при возврате в меню из паузы как `abandoned`.
- Итог сессии:
  - `defeat` при смерти игрока,
  - `victory` при победе над боссом,
  - `abandoned` при выходе в меню до завершения цикла.

### 3) Правило принятия решений по итогам 5–10 сессий
1. Снять выгрузку из `localStorage` (ключ `ashfang-playtest-runs`).
2. Посчитать медиану time-to-fun и распределение исходов.
3. Сопоставить видео/заметки наблюдений с числовыми всплесками (например, высокий `pauseCount`).
4. Сформировать top-3 исправлений только по частоте повторяемости проблемы.

## Как проверить реализацию
1. Запустить игру (`npm run dev`).
2. Пройти короткую сессию: старт → бой → пауза → выход/победа/поражение.
3. Открыть DevTools → Application → Local Storage.
4. Проверить ключ `ashfang-playtest-runs`: в массиве появляется новая запись со структурой:
   - `outcome`,
   - `durationMs`,
   - `timeToFunMs`,
   - `pauseCount`,
   - `menuExitCount`.

## Критерий готовности DEV-36
- Есть формализованный цикл наблюдаемого playtest.
- Есть рабочий сбор базовых поведенческих метрик по каждой попытке.
- Есть правило приоритизации правок по частоте проблемы, а не по субъективному ощущению.
