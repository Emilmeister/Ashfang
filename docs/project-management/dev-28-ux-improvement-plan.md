# DEV-28 — План улучшения пользовательского опыта (UX) для MVP

- Статус: draft
- Связь с Linear: `DEV-28`
- Формат: **«Что есть сейчас → Что должно быть в хорошем продукте»**
- Цель документа: превратить практики из геймдев-чеклиста в конкретный план улучшений для `Ashfang: Echoes of the Fallen Wild`.

## 1) Core loop и корневая проблема

**Что есть сейчас**
- Базовый цикл уже реализован: «войти в уровень → победить врагов → открыть портал → бой с боссом».  
- Цель в уровне показана через HUD-текст и onboarding-подсказки.

**Что должно быть в хорошем продукте**
- Core loop должен быть формализован и измерим:  
  **«Игрок двигается и ищет окно для атаки → получает ощутимый боевой фидбэк и прогресс по цели → принимает следующее решение по позиционированию/риску»**.
- Для команды loop должен иметь KPI:  
  - time-to-fun < 30 сек,
  - понятность цели без внешних объяснений,
  - отсутствие «пустых» сегментов > 5 секунд.

---

## 2) Усиление базового gameplay (движение/атака)

**Что есть сейчас**
- Есть движение (WASD/стрелки), атака по `SPACE`, кулдауны, урон и HP.
- Бой функциональный, но с ограниченной глубиной (одна базовая атака, один базовый ритм столкновения).

**Что должно быть в хорошем продукте**
- Каждое базовое действие должно давать «удовольствие от нажатия»:
  - движение: короткий рывок/ускорение или micro-smoothing,
  - атака: более читаемая анимационная фаза (startup/impact/recover),
  - попадание: более явный импакт и реакция цели.
- Добавить хотя бы 1 дополнительное тактическое действие в MVP (например, short-dash с кулдауном) для создания микровыбора в бою.

---

## 3) Мгновенный фидбэк (juice)

**Что есть сейчас**
- Уже есть базовые SFX (атака/получение урона/подтверждение UI), tint/tween-эффекты и отдельные статусы на HUD.

**Что должно быть в хорошем продукте**
- Усилить ощущение действия на каждом hit:
  - короткий screen shake на попадание,
  - hit flash + частицы,
  - реакция врага (knockback/micro-stun или явный hit-react),
  - короткий критический звук/визуал при добивании.
- Для каждого боевого события должна быть связка «визуал + звук + изменение состояния», чтобы игрок не сомневался, что произошло.

---

## 4) Решения и риск/награда

**Что есть сейчас**
- Игрок в основном повторяет цикл «подойти и атаковать», тактические альтернативы ограничены.

**Что должно быть в хорошем продукте**
- Добавить минимум 2 боевых решения:
  - безопасная стратегия (держать дистанцию/кайтить),
  - рискованная стратегия (вход в ближний бой ради более быстрого результата).
- Добавить ресурс, который заставляет выбирать (например, заряд рывка, временное окно усиленного удара или ограниченный heal pickup).

---

## 5) Прогрессия, которая меняет геймплей

**Что есть сейчас**
- Прогрессия в основном структурная: переход «уровень → босс».
- Нет явного ощущения усиления персонажа внутри сессии.

**Что должно быть в хорошем продукте**
- Ввести короткую сессионную прогрессию MVP:
  - после мини-цели игрок получает модификатор (например, +скорость, +дальность атаки, hit-wave),
  - модификатор меняет стиль прохождения, а не только цифры HP.
- Игрок должен почувствовать «я стал сильнее/иначе играю» до входа в босс-файт.

---

## 6) Вариативность и неожиданность

**Что есть сейчас**
- Поведение и состав столкновений в основном предсказуемы.

**Что должно быть в хорошем продукте**
- Добавить легкую вариативность без раздувания scope:
  - 2-й тип врага с другим паттерном,
  - случайный модификатор комнаты/волны (например, «туман», «быстрые враги», «усиленный урон игрока»),
  - небольшая рандомизация спавна из подготовленных точек.
- Каждая новая попытка должна отличаться хотя бы 1-2 моментами.

---

## 7) Удаление скучных промежутков

**Что есть сейчас**
- Есть перемещения между зонами и сегменты без явного действия.

**Что должно быть в хорошем продукте**
- Правило для MVP: если >5 секунд без решения/угрозы/цели — добавляем событие.
- Сократить «пустой бег»:
  - компактнее layout критического пути,
  - микро-ивенты в переходах (малые засады, интерактивные объекты, реплики),
  - визуальные ориентиры, чтобы игрок не терял темп.

---

## 8) Playtesting как основной источник решений

**Что есть сейчас**
- В roadmap playtest уже предусмотрен как отдельная задача, но без формализованных UX-метрик на этот issue.

**Что должно быть в хорошем продукте**
- Ввести цикл наблюдаемого плейтеста на DEV-28:
  - 5–10 сессий,
  - фиксируем точки: где игрок теряется, где скучает, где получает урон «непонятно почему»,
  - собираем поведенческие метрики: time-to-fun, время до первой смерти/победы, частота паузы/выхода.
- Приоритизировать правки по частоте проблемы у игроков, а не по субъективному ощущению разработчика.

---

## 9) Быстрые итерации

**Что есть сейчас**
- Улучшения UX пока не декомпозированы в короткие гипотезы и проверяемые батчи.

**Что должно быть в хорошем продукте**
- Работать спринтами по 1-2 дня:
  - гипотеза (например, «screen shake улучшит читаемость hit»),
  - изменение,
  - мини-плейтест,
  - вывод (оставить/откатить/переосмыслить).
- Каждый UX-коммит должен иметь критерий проверки (что именно должно улучшиться для игрока).

---

## 10) Упростить лишнее

**Что есть сейчас**
- Базовая структура уже достаточно компактная, но есть риск наращивать фичи вместо усиления ядра.

**Что должно быть в хорошем продукте**
- Для MVP фиксируем принцип: **удаляем/не добавляем все, что не усиливает core loop**.
- Любая новая механика проходит фильтр:
  1. Улучшает ли «удовольствие от базового действия»?
  2. Добавляет ли meaningful choice?
  3. Уменьшает ли скучные промежутки?
- Если ответ «нет» хотя бы по двум пунктам — механику откладываем post-MVP.

---

## Быстрый чек-лист качества (для acceptance DEV-28)

Считаем задачу выполненной, когда:
- Игрок получает «fun» в первые 30 секунд.
- Цель уровня понятна без внешних объяснений.
- В бою есть минимум 2 осмысленных решения.
- На каждое важное действие есть мгновенный фидбэк (звук + визуал + реакция состояния).
- Пустые интервалы >5 секунд убраны или заполнены интерактивом.
- Результаты подтверждены плейтестом, а не только внутренней оценкой.

## Предлагаемые подзадачи внутри DEV-28

1. Усилить боевой juice (shake, hit VFX, реакция врага).
2. Добавить 1 тактическое действие игрока (dash/аналог).
3. Добавить 2-й тип врага или модификатор столкновений.
4. Внедрить краткую сессионную прогрессию до босса.
5. Сократить «пустые» сегменты уровня по правилу 5 секунд.
6. Провести UX-playtest и приоритизировать backlog правок по наблюдениям.
