# DEV-29 — Core loop и корневая проблема

- Статус: draft
- Связь с Linear: `DEV-29`
- Формат: **«Что есть сейчас → Что должно быть в хорошем продукте»**
- Цель: зафиксировать корневую UX-проблему и перевести core loop в измеримый цикл для MVP.

## Формулировка core loop (целевая)

**Игрок двигается и ищет окно для атаки → получает ощутимый боевой фидбэк и прогресс по цели → принимает следующее решение по позиционированию/риску.**

---

## 1) Корневая проблема: loop есть, но он слабо измерим

**Что есть сейчас**
- Игровая последовательность работает: «уровень → враги → портал → босс».
- Цель формально показана через HUD и onboarding.
- UX-оценка loop в основном качественная (по ощущениям), без единого набора KPI.

**Что должно быть в хорошем продукте**
- Core loop описан одной фразой и принят командой как общий ориентир.
- Для loop зафиксированы базовые KPI:
  - `time-to-fun < 30 сек`;
  - игрок понимает цель без внешних объяснений;
  - нет «пустых» сегментов > 5 секунд.
- Каждая итерация DEV-29 должна проверяться по этим KPI, а не только по субъективному «стало лучше».

---

## 2) Прозрачность цели и направляющий UX

**Что есть сейчас**
- Информация о цели есть, но опирается на текстовые подсказки.
- При первом заходе игрок может читать, а не действовать.

**Что должно быть в хорошем продукте**
- Цель считывается по визуальному контексту и поведению мира (HUD — как подтверждение, а не единственный источник).
- Игрок в первые 10–15 секунд понимает:
  1. где опасность,
  2. что нужно делать,
  3. какой ближайший шаг приведёт к прогрессу.
- Подсказки короткие, контекстные и не тормозят начало действия.

---

## 3) Непрерывность действия (правило «без пустоты >5 сек»)

**Что есть сейчас**
- Между столкновениями возможны интервалы без решения/угрозы.
- Темп может проседать на переходах или в «пустом беге».

**Что должно быть в хорошем продукте**
- Любой интервал >5 секунд без решения игрока считается дефектом темпа.
- Переходы заполняются микро-событиями: риск, награда, мини-выбор, читаемый ориентир.
- Даже вне боя игрок регулярно принимает решение: войти в риск, обойти, добрать ресурс, ускорить путь к цели.

---

## 4) Фидбэк как часть loop, а не косметика

**Что есть сейчас**
- Урон, атака и победа функционально работают.
- Фидбэк местами информативен, но не всегда «ощутим» по импакту.

**Что должно быть в хорошем продукте**
- Каждое ключевое действие (удар, получение урона, добивание, открытие портала) даёт мгновенную связку:
  - визуальная реакция,
  - звуковая реакция,
  - изменение состояния/прогресса.
- Игрок должен без интерфейсного «декодирования» чувствовать: «попал / не попал», «риск окупился / не окупился».

---

## 5) Решение игрока как центр цикла

**Что есть сейчас**
- Есть базовое «подойти и ударить», но риск/награда ограничены.
- Повторяемость действий может снижать вовлечённость.

**Что должно быть в хорошем продукте**
- Внутри одной боевой ситуации у игрока минимум 2 осмысленных опции (агрессивный/безопасный стиль, короткий риск ради ускорения прогресса и т.п.).
- Повтор одного и того же действия не должен быть оптимальным «всегда и везде».
- Решение игрока должно напрямую влиять на темп и выживаемость.

---

## KPI для DEV-29 (MVP-уровень)

- **TTF (time-to-fun):** первый «осмысленный боевой момент» ≤ 30 сек от старта уровня.
- **Goal clarity:** не менее 80% новых игроков корректно формулируют цель уровня после первой попытки.
- **Downtime:** 0 сегментов > 5 секунд без действия/решения.
- **Loop confidence:** игрок в плейтесте может словами описать цикл «действие → фидбэк/прогресс → следующее решение».

## Как проверять на плейтесте

1. 5–10 коротких сессий с наблюдением без подсказок.
2. Тайминг-замеры: старт уровня, первый контакт, первый kill, открытие портала.
3. Лог проблем по трём категориям:
   - непонятная цель,
   - слабый фидбэк,
   - провалы темпа (>5 сек).
4. После каждой итерации — решение: оставить, откатить или доработать гипотезу.

## Definition of Done для DEV-29

- Core loop утверждён как единая формулировка для команды.
- KPI задокументированы и применяются в проверке изменений.
- Есть подтверждение на плейтесте, что:
  - цель понятна без внешних объяснений,
  - первые 30 секунд дают «вход в fun»,
  - длинные пустые сегменты устранены.


## Реализация в текущем MVP (инкремент DEV-29)

- В `LevelScene` добавлены runtime-метрики loop:
  - отображение TTF (time-to-fun) как времени до первого подтверждённого боевого попадания;
  - отображение времени сессии на HUD.
- Цель в HUD сделана динамической: показывается оставшееся число врагов до открытия портала.
- Добавлен анти-downtime механизм: при паузе в meaningful action >5 секунд до открытия портала появляется «pressure enemy», чтобы вернуть игрока в решение/риск.
