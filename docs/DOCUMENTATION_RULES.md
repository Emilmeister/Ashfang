# Правила ведения документации для проекта 2D браузерной игры

## 1. Цели документации

Документация должна:
- ускорять онбординг новых участников команды;
- снижать bus factor для ключевых подсистем;
- фиксировать архитектурные и продуктовые решения;
- обеспечивать единый язык между разработкой, гейм-дизайном, QA, DevOps и аналитикой.

**Ключевой принцип:** документация должна быть "живой" — обновляться одновременно с изменениями в коде/процессах.

---

## 2. Общие принципы

1. **Single source of truth**
   - У каждого типа информации должен быть один канонический документ.
   - Ссылки на внешние источники допустимы, но основной контекст должен оставаться в репозитории.

2. **Docs-as-code**
   - Документация хранится в Git рядом с кодом.
   - Изменения в документации проходят через PR и review.

3. **Измеримость и актуальность**
   - У каждого важного документа должен быть владелец (owner).
   - У документа должен быть раздел: `Последняя проверка` и `Следующий срок ревизии`.

4. **Понятность и минимализм**
   - Писать короткими абзацами, использовать списки и таблицы.
   - Избегать двусмысленных формулировок и неочевидных сокращений.

---

## 3. Обязательная структура документации в репозитории

```text
docs/
  README.md                       # Карта документации
  product/                        # Игровая логика, механики, экономика
  architecture/                   # Архитектура, схемы, ADR
  backend/                        # API, сервисы, контракты
  frontend/                       # UI, state management, клиентская архитектура
  game-design/                    # Баланс, прогрессия, игровые циклы
  operations/                     # CI/CD, релизы, мониторинг, инциденты
  qa/                             # Тест-стратегия, чек-листы, тестовые матрицы
  security/                       # Модель угроз, политики доступа
  glossary.md                     # Термины проекта
  changelog-docs.md               # История значимых изменений документации
  templates/                      # Шаблоны документов
```

Если раздела пока нет, создается `README.md` с пометкой "в разработке" и ответственным.

## 4. Категории документов и требования

### 4.1 Product и Game Design
- Описание core loop и meta loop.
- Спецификации механик в формате: цель, правила, параметры, edge cases.
- Экономические таблицы (валюты, источники, sinks, ограничения).
- Синхронизация с аналитикой: перечень событий и KPI.

### 4.2 Architecture
- Диаграмма контекста системы.
- Диаграммы ключевых потоков (авторизация, матчмейкинг, бой, награды).
- ADR (Architecture Decision Records) для всех решений с влиянием на масштабирование, производительность, безопасность и стоимость.

### 4.3 Backend / Frontend
- Контракты API (endpoint, request/response, ошибки, ограничения).
- Версионирование API и правила обратной совместимости.
- Документация по состояниям клиента, кэшированию, ретраям, обработке ошибок.

### 4.4 Operations / DevOps
- Runbook запуска среды (локальной, staging, production).
- Release checklist.
- Incident response playbook (классификация инцидентов, SLA реакции, каналы эскалации).

### 4.5 QA
- Тестовая стратегия: уровни тестирования и зоны ответственности.
- Матрица регрессии для критичных сценариев.
- Правила оформления баг-репортов.

### 4.6 Security
- Модель угроз для клиент-серверного взаимодействия.
- Защита игровых событий от мошенничества (валидация на сервере, rate limit, античит-механики).
- Политика работы с секретами и правами доступа.

## 5. Единый формат каждого документа

Каждый значимый документ должен начинаться с метаблока:

```md
# Название

- Статус: draft | active | deprecated
- Владелец: @team или @username
- Последняя проверка: YYYY-MM-DD
- Следующий срок ревизии: YYYY-MM-DD
- Связанные артефакты: [ссылки на issue/PR/дашборды]
```

Далее:
1. Контекст
2. Цель
3. Основное содержание
4. Ограничения и риски
5. Критерии готовности/актуальности
6. История изменений

---

## 6. Процесс обновления документации

1. Любой PR, изменяющий поведение системы, **обязан** содержать:
   - обновление релевантной документации, или
   - явное объяснение, почему обновление не требуется.

2. В PR используется чек-лист:
   - [ ] документация обновлена;
   - [ ] добавлены/обновлены схемы (если есть архитектурные изменения);
   - [ ] обновлены API-контракты (если есть изменения интерфейсов);
   - [ ] указан владелец новых документов.

3. Для крупных изменений — отдельный doc-first PR до реализации.

4. Раз в спринт проводится ревизия docs backlog:
   - устаревшие документы;
   - документы без владельца;
   - пробелы в критичных знаниях.

---

## 7. Версионирование и устаревание

- Изменения в документации фиксируются в `docs/changelog-docs.md`.
- При устаревании документ помечается `deprecated`, но не удаляется до миграции ссылок.
- Для критичных документов (архитектура, безопасность, runbook) ревизия не реже 1 раза в 3 месяца.

---

## 8. Стандарты качества документации

Документ считается качественным, если он:
- **Актуален**: соответствует текущему коду и процессам.
- **Проверяем**: содержит конкретные шаги/критерии, а не общие слова.
- **Навигируем**: имеет оглавление и перекрестные ссылки.
- **Воспроизводим**: по инструкции можно повторить процесс без устных пояснений.

Рекомендуется использовать:
- Markdown + Mermaid для диаграмм.
- Единые именования файлов: `kebab-case.md`.
- Явные даты в формате ISO (`YYYY-MM-DD`).

---

## 9. Роли и ответственность

- **Tech Lead**: архитектурная документация и ADR.
- **Backend/Frontend Leads**: контракты и технические гайды по своим областям.
- **Game Designer**: документация механик и экономики.
- **QA Lead**: тестовая стратегия и регрессия.
- **DevOps/SRE**: runbook, релизы, мониторинг, инциденты.
- **Engineering Manager / Project Lead**: контроль полноты и регулярности ревизий.

---

## 10. Минимальный стартовый набор (MVP) для проекта

Для запуска процесса в первую очередь должны быть созданы:
1. `docs/README.md` — карта документации.
2. `docs/architecture/system-overview.md` — обзор архитектуры.
3. `docs/backend/api-contract.md` — базовый API-контракт.
4. `docs/game-design/core-loop.md` — описание core loop.
5. `docs/operations/release-checklist.md` — чек-лист релиза.
6. `docs/operations/incident-playbook.md` — базовый incident playbook.
7. `docs/glossary.md` — единый словарь терминов.

Это обязательный минимум, без которого проект считается рисковым с точки зрения передачи знаний.
